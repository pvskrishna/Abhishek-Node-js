## 📋 TABLE OF CONTENTS

### **CREATIONAL PATTERNS** (Object Creation)
1. **Singleton** - Only one instance of a class
2. **Factory Method** - Create objects without specifying exact classes
3. **Builder** - Construct complex objects step-by-step

### **STRUCTURAL PATTERNS** (Object Composition)
4. **Adapter** - Make incompatible interfaces work together
5. **Decorator** - Add behavior to objects dynamically
6. **Facade** - Simplify complex subsystems
7. **Proxy** - Control access to another object
8. **Composite** - Treat individual objects and compositions uniformly

### **BEHAVIORAL PATTERNS** (Object Collaboration)
9. **Observer** - Notify multiple objects when state changes
10. **Strategy** - Encapsulate interchangeable algorithms
11. **Command** - Encapsulate requests as objects
12. **Iterator** - Access elements sequentially without exposing structure
13. **State** - Alter behavior based on internal state
14. **Template Method** - Define algorithm skeleton, subclasses override details
15. **Chain of Responsibility** - Pass request along chain of handlers

## 💡 PATTERN SELECTION GUIDE

### **Problem: How do I manage object creation?**
→ Use: **Singleton** (one instance) or **Factory** (abstract creation) or **Builder** (complex objects)

### **Problem: How do I adapt two incompatible interfaces?**
→ Use: **Adapter**

### **Problem: How do I add behavior without modifying code?**
→ Use: **Decorator**

### **Problem: How do I simplify a complex subsystem?**
→ Use: **Facade**

### **Problem: How do I notify multiple objects of changes?**
→ Use: **Observer** or **Observer Pattern**

### **Problem: How do I swap algorithms at runtime?**
→ Use: **Strategy**

### **Problem: How do I undo/redo actions?**
→ Use: **Command**

### **Problem: How do I handle requests in a chain?**
→ Use: **Chain of Responsibility**

### **Problem: How do I handle complex state transitions?**
→ Use: **State** or **State Machine**

## 🔍 COMMON INTERVIEW QUESTIONS (ACROSS ALL PATTERNS)

**Q1: "What are design patterns?"**
→ Reusable solutions to common programming problems, proven approaches

**Q2: "Why use design patterns?"**
→ Code reusability, maintainability, scalability, communication with team

**Q3: "What's the difference between patterns and principles?"**
→ Principles (SOLID) guide decisions, patterns implement principles

**Q4: "Can patterns have negative impact?"**
→ Yes - over-engineering, unnecessary complexity, performance overhead

**Q5: "How do you decide which pattern to use?"**
→ Analyze problem → identify constraints → match pattern → evaluate trade-offs
